## Architecture

Fully client-side React 19 + Vite app — no backend, no API calls, no server. All persistent state lives in `localStorage` under the key `co:expense_mapping_rules`.

**Routing** is file-based via TanStack Router. `src/routeTree.gen.js` is auto-generated by `@tanstack/router-plugin` during `vite dev`/`build` — never edit it manually. Add routes by creating files under `src/routes/`.

**Tailwind v4** is configured via the `@tailwindcss/vite` plugin — there is no `tailwind.config.js`.

## Core data model

When a CSV is parsed, PapaParse injects two synthetic fields into every transaction row:
- `CO_id` — integer index used as the React list key and for targeting rows on category change
- `CO_category` — empty string initially; filled in during categorization

The **identifier** is the column the user selects as the payee/payer field. Mapping rules are stored as `{ [transaction[identifier]]: categoryString }`.

Subcategories use `::` as a delimiter (e.g. `Expense::Grocery`). On export in `FinishCategorization`, `CO_category` is split on `::` and written out as `CO_category`, `CO_category_2`, etc. into the downloaded CSV. `CO_id` is stripped before export.

## Categorization wizard (`/categorize`)

The wizard is a step-based UI in `src/routes/categorize.jsx`. The step order differs depending on whether the user has existing mapping rules:

- **New user**: ImportStatement → TransactionTable → ChooseIdentifier → FinishCategorization
- **Returning user**: ImportStatement → ChooseIdentifier → TransactionTable → FinishCategorization

Each step component receives `previousStep` and `nextStep` callbacks. `FinishCategorization` auto-triggers CSV download and saves new mapping rules to localStorage via `useEffect` on mount.

## Mapping rules utility (`src/utils/mappingRules.js`)

Four exported functions for reading/writing/merging/clearing `localStorage`. `addMappingRulesToLocalStorage` merges (new rules overwrite existing ones for the same identifier value).

## Feature design reference

When discussing or implementing new features, benchmark against how established expense tracking apps handle the same problem:

- **YNAB** — gold standard for zero-based budgeting UX; reference for category management, rule-based assignment, and goal tracking
- **Monarch Money** — modern Mint replacement; reference for transaction review flow, tagging, and split transactions
- **Copilot** — Apple-native polish; reference for merchant enrichment, recurring detection, and visual spend summaries
- **Mint** (sunset Jan 2024) — historical reference for aggregation UX patterns now widely adopted elsewhere

Before suggesting a feature, consider how these apps solved the same problem and whether CategorizeOnce should match, simplify, or intentionally diverge from their approach.

## Prebuild scripts

`npm run build` auto-runs two Node scripts first:
- `scripts/generate-attributions.js` — regenerates `THIRD-PARTY-NOTICES.txt`
- `scripts/add-license-headers.js` — adds Apache 2.0 license headers to source files that are missing them

All new `.jsx`/`.js` source files should include the Apache 2.0 license header block (see any existing file for the template).
